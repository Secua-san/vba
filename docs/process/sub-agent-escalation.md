# Sub-Agent Escalation

## 目的
- 親エージェントが実装責任に集中し、非実装作業を抱え込まない
- 思考スタックや探索の空転を短く切り上げる
- 主担当の思い込みを減らし、別視点の確認を入れる
- 親エージェントを整理担当ではなく実装責任者に戻し続ける

## サブエージェントへ切り替える目安
- 優先度確認、branch / main / 作業状態確認、1 PR slice 比較、既存パターン探索、再現確認、整理系 skill 実行要否判定、実装差分 review のような非実装作業が必要
- 親エージェントが 2〜3 ステップ以上連続で調査だけを続けそう
- 同じ設計論点を 2 回以上繰り返し検討している
- 追加の検索や読解をしても、新しい根拠が増えていない
- 実装方針を決める前にコードベースの事実確認が必要
- 複数の妥当な案があり、優先順位づけが進まない

## 標準役割
旧 `impact-investigator` は廃止し、役割を次の subagent へ分割する。

### `task-priority-auditor`
- 目的: `PLAN.md` / `TASKS.md` / issue / TODO / 現在の作業状態から、優先度の高い未完了タスク候補を短く返す
- 担当: 優先タスク候補の抽出、未完了状態の確認、依存関係の確認、すぐ着手できるかの確認
- 禁止: 実装計画の長文化、実装の代行、ドキュメント整理、方針文書の再構成
- 成果物: 優先候補 1〜3 件、その理由、着手前提の不足有無

### `branch-state-checker`
- 目的: 現在のブランチ、`main` との関係、未コミット差分、実装着手前提を確認する
- 担当: 現在ブランチの確認、`main` との差分状態の確認、未コミット変更や競合リスクの確認、実装開始可否の判定補助
- 禁止: 実装内容の決定、ドキュメント更新、長文講評
- 成果物: 現在の作業状態、実装開始可否、注意点

### `slice-scout`
- 目的: 複数の実装候補から、1 PR に収まりやすい最小 slice を比較・提案する
- 担当: `semantic token` / `diagnostics` / `symbol` / `formatter` / `documentService` などの候補比較、変更範囲の概算、既存依存の強さの確認、1 PR に収まりやすい粒度の提案
- 禁止: 実装の代行、大規模再設計、長文の方針整理
- 成果物: 候補ごとの比較、最小 slice の提案、避けるべき大きすぎる切り口

### `pattern-investigator`
- 目的: 既存実装のパターンや依存箇所を短く調査する
- 担当: `documentService` / `formatter` / `semantic token` / `diagnostics` / `symbol` まわりの既存パターン調査、regex 依存・line 走査依存・text 判定残存箇所の洗い出し、呼び出し関係の把握
- 禁止: 実装の代行、方針文書の整理、調査結果の長文化
- 成果物: 関連ファイル、関連箇所、注意すべき依存、既存パターン要約

### `repro-prober`
- 目的: ごく小さい再現ケースで現状の挙動を確認する
- 担当: multiline header の local variable / built-in / member の現状確認、`semantic token` / `diagnostics` の不足確認、切り口確定のための最小再現
- 禁止: 実装修正、広範囲テスト整備、ドキュメント更新
- 成果物: 再現手順、観測結果、実装前に押さえるべき不足点

### `skill-gatekeeper`
- 目的: `.codex/skills/doc-minimum-update` と `.codex/skills/lightweight-review` を実行してよいかだけ判定する
- 担当: `doc-minimum-update` の実行要否判定、`lightweight-review` の実行要否判定
- 禁止: 実際の大規模整理、抽象的な方針文書作成、実装の差し戻し前提の講評
- 成果物: 実行可 / 不要、対象 skill、理由 1〜3 行

### `diff-reviewer`（既定実体: `reviewer`）
- 目的: 実装差分に対する軽量レビューだけを行う
- 担当: 回帰、境界条件、テスト不足、不要差分の観点での確認
- 禁止: 実装の差し戻しを前提にした大規模再設計、全面レビュー、設計思想の再整理
- 成果物: 修正すべき可能性のある箇所、軽微な改善提案、ブロッカーの有無

## このリポジトリでの役割制約
- 通常タスクの進め方と Done の定義は [../../AGENTS.md](../../AGENTS.md) を正本とし、サブエージェントはその補助に限って使う
- 主担当が常に実装責任を持ち、サブエージェントは調査、優先度確認、作業状態確認、slice 選定、再現確認、整理系 skill の実行可否判定、差分レビューの補助に限って使う
- サブエージェントの成果は短く受け取り、最終的な実装対象の決定、コード変更、検証、完了判定は必ず親エージェントが行う
- 長い方針文書の整理、抽象的な指針策定、実装の代行を主作業としてサブエージェントへ委譲しない
- 親エージェントが調査そのものを成果物にしない。実装可能なら調査を打ち切ってコード変更へ進む
- 調査結果や判定結果を受けたら、その作業単位の中で親エージェントが必ずコード変更、必要な検証、必要最小限の追記まで進める
- skill やサブエージェントの成果だけで完了しない
- サブエージェントの結果待ちだけで止まらず、親エージェントは並行して非重複の準備や実装判断を進める

## 依頼の出し方
- 論点を 1 つに絞る
- 対象ファイル、前提、迷っている選択肢を明示する
- 期待する出力を短く指定する
- 広すぎる丸投げをしない

## 親エージェントの責務
- タスクの主目的を実装と定義する
- 優先度確認、slice 比較、再現確認などの非実装作業では、まず対応する subagent を呼び出すことを検討する
- subagent の結果を受けて自ら実装対象を決め、コード変更を実施する
- 必要な検証を行う
- 必要な場合のみ `.codex/skills/` 配下の skill を実行する
- 最終的な完了判定を行う

## 標準フロー
1. 親エージェントが実装目的を短く定義する。
2. `task-priority-auditor` に優先候補確認を依頼する。
3. 必要なら `branch-state-checker` に着手前提を確認させる。
4. 必要なら `slice-scout` に 1 PR slice 比較を依頼する。
5. 必要なら `pattern-investigator` に既存依存を調査させる。
6. 必要なら `repro-prober` に最小再現確認をさせる。
7. 親エージェントが実装対象を決定する。
8. 親エージェントがコード変更を行う。
9. 必要なテスト、型チェック、lint、build、最小動作確認を行う。
10. 必要なら `skill-gatekeeper` で整理系 skill の実行要否を判定する。
11. 必要な場合だけ `.codex/skills/doc-minimum-update` または `.codex/skills/lightweight-review` を実行する。
12. 必要なら `diff-reviewer` で軽量レビューを行う。
13. 必要なら `.codex/skills/task-close-check` で完了条件を確認する。
14. 親エージェントが完了を宣言する。

補足:
- すべての subagent を毎回使う必要はない
- ただし、優先度確認、slice 比較、再現確認のような非実装作業を親エージェントが抱え込み続けない
- 実装可能なら調査を打ち切ってコード変更へ進む

## PR 前の必須レビュー
- PR 作成前に、主担当とは別視点のサブエージェントレビューを最低 1 回実施する
- 既定の担当は `diff-reviewer` 役割（実体は `reviewer`）とし、確認観点は「回帰」「境界条件」「テスト不足」「不要差分」とする
- `reviewer` が使えない場合は設定不整合として扱い、`C:\Users\tagi0\.codex\config.toml` と `C:\Users\tagi0\.codex\agents\reviewer.toml` を確認してから作業を再開する
- 指摘が 0 件でも結果を要約してから PR 本文作成に進む
- 重大な指摘が残る場合は PR を作成せず、先に修正する
- PR 前レビューの指摘と CodeRabbit の指摘が同一論点で重なった場合は、`docs/process/coderabbit-review.md` の重複指摘ルールに従って原則修正する
- `required` / `optional` のような運用上の挙動差の判断基準も、`docs/process/coderabbit-review.md` を正本として扱う

## 避けること
- 非実装作業を親エージェントが抱え込んだまま 2〜3 ステップ以上連続で調査し続ける
- 実装対象が決まっているのに、優先度確認や再現確認を延々と繰り返す
- 複数論点を一度に投げて、責務を曖昧にする
- サブエージェントの返答を無検証でそのまま採用する
- 実装前に整理系 skill の起動要否をサブエージェントへ広げる
