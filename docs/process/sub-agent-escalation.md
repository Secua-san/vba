# Sub-Agent Escalation

## 目的
- 思考スタックや探索の空転を短く切り上げる
- 主担当の思い込みを減らし、別視点の確認を入れる
- 親エージェントを整理担当ではなく実装責任者に戻し続ける

## サブエージェントへ切り替える目安
- 同じ設計論点を 2 回以上繰り返し検討している
- 追加の検索や読解をしても、新しい根拠が増えていない
- 実装方針を決める前にコードベースの事実確認が必要
- 複数の妥当な案があり、優先順位づけが進まない

## 標準役割
- `impact-investigator`（既定実体: `explorer`）: 関連ファイル、既存実装、影響範囲、依存関係の調査だけを行う
- `impact-investigator` の成果物: 変更対象候補、影響範囲、注意点、既存パターンの要約
- `impact-investigator` の禁止: 実装の代行、長文化した実装計画、指針文書の整理、ドキュメント更新の主導
- `skill-gatekeeper`（既定実体: `default`）: `.codex/skills/doc-minimum-update` または `.codex/skills/lightweight-review` を実行してよいかだけを判定する
- `skill-gatekeeper` の判定基準: 実装が完了しているか同一タスク内で実装完了が確実か、対象が実装差分に直接関係するか、変更が必要最小限か、実行しないと説明不足や手順不整合が出るか、整理が主作業化していないか
- `skill-gatekeeper` の成果物: 実行可 / 実行不要、対象 skill 名、理由 1〜3 行
- `skill-gatekeeper` の禁止: 大規模な文書整理、抽象的方針文書の作成、実装の差し戻し前提の長文講評、実装前の整理拡大
- `diff-reviewer`（既定実体: `reviewer`）: 実装差分に対する簡易レビューだけを行う
- `diff-reviewer` の成果物: 修正すべき可能性のある箇所、軽微な改善提案、ブロッカーの有無
- `diff-reviewer` の禁止: 実装の差し戻しを前提にした大規模再設計、全面レビュー、設計思想の再整理

## このリポジトリでの役割制約
- 通常タスクの進め方と Done の定義は [../../AGENTS.md](../../AGENTS.md) を正本とし、サブエージェントはその補助に限って使う
- 主担当が常に実装責任を持ち、サブエージェントは調査、整理系 skill の実行可否判定、差分レビューの補助に限って使う
- 長い方針文書の整理、抽象的な指針策定、実装の代行を主作業としてサブエージェントへ委譲しない
- 調査結果や判定結果を受けたら、その作業単位の中で親エージェントが必ずコード変更、必要な検証、必要最小限の追記まで進める
- skill やサブエージェントの成果だけで完了しない

## 依頼の出し方
- 論点を 1 つに絞る
- 対象ファイル、前提、迷っている選択肢を明示する
- 期待する出力を短く指定する
- 広すぎる丸投げをしない

## 親エージェントの責務
- タスクの主目的を実装と定義する
- 必要に応じて subagent を呼び出す
- subagent の結果を受けて自らコード変更を実施する
- 必要な検証を行う
- 必要な場合のみ `.codex/skills/` 配下の skill を実行する
- 最終的な完了判定を行う

## 標準フロー
1. 親エージェントがタスクを実装中心に再定義する。
2. 必要なら `impact-investigator` で影響範囲を調査する。
3. 親エージェントが最小実装を行う。
4. 必要なテスト、型チェック、lint、build、最小動作確認を行う。
5. 迷う場合だけ `skill-gatekeeper` で整理系 skill の実行要否を判定する。
6. 必要な場合のみ `.codex/skills/doc-minimum-update` を実行する。
7. 必要なら `diff-reviewer` または `.codex/skills/lightweight-review` を実行する。
8. `.codex/skills/task-close-check` で完了条件を確認する。
9. 親エージェントが完了を宣言する。

## PR 前の必須レビュー
- PR 作成前に、主担当とは別視点のサブエージェントレビューを最低 1 回実施する
- 既定の担当は `diff-reviewer` 役割（実体は `reviewer`）とし、確認観点は「回帰」「境界条件」「テスト不足」「不要差分」とする
- `reviewer` が使えない場合は設定不整合として扱い、`C:\Users\tagi0\.codex\config.toml` と `C:\Users\tagi0\.codex\agents\reviewer.toml` を確認してから作業を再開する
- 指摘が 0 件でも結果を要約してから PR 本文作成に進む
- 重大な指摘が残る場合は PR を作成せず、先に修正する
- PR 前レビューの指摘と CodeRabbit の指摘が同一論点で重なった場合は、`docs/process/coderabbit-review.md` の重複指摘ルールに従って原則修正する
- `required` / `optional` のような運用上の挙動差の判断基準も、`docs/process/coderabbit-review.md` を正本として扱う

## 避けること
- まだ調べていない基本情報の代行として毎回使う
- 複数論点を一度に投げて、責務を曖昧にする
- サブエージェントの返答を無検証でそのまま採用する
- 実装前に整理系 skill の起動要否をサブエージェントへ広げる
