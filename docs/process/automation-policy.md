# Automation Policy

## 品質ゲート
- `npm run lint`
- `npm run build`
- `npm test`
- 必要に応じて `npm run package`

## 品質ゲートの原則
- Windows ネイティブ環境で build / test / package が通ることを優先する
- 品質ゲートが失敗した状態では自動コミットしない
- 失敗理由が不明なまま自動 PR を作成しない
- 自動化を止める場合は、失敗コマンド、失敗理由、想定影響、人間確認が必要な点を要約する

## 自動コミット許可条件
- 変更が論理単位で分割されている
- 品質ゲート違反がない
- 機密情報を含まない
- 破壊的変更ではない

## 自動 PR 許可条件
- 自動コミット許可条件を満たしている
- 仕様が明確である
- 低リスクである
- 差分がレビュー可能サイズである
- 禁止領域に該当しない

## 自動 PR 禁止領域
- 認証
- 権限管理
- 課金
- シークレット
- CI/CD 設定
- インフラ
- DB 構造変更
- マイグレーション
- 依存関係メジャー更新
- 破壊的変更
- データ消失の可能性がある変更

## 停止条件
- 仕様が曖昧
- 差分が大きすぎる、または複数目的で混在している
- テスト失敗や build 失敗の原因が不明
- リポジトリ状態が壊れている
- 既存方針や要件と衝突する
- 変更意図を 1 文で説明できない

## リポジトリ作成を許可する条件
- プロジェクトが未初期化
- 大規模機能を独立開発する必要がある
- モノレポ分割が必要
- 実験的機能を隔離する必要がある

## リポジトリ作成時のルール
- GitHub 上に新規リポジトリを作成する
- 初期テンプレート構成を生成する
- `README` / `LICENSE` / `.gitignore` を生成する
- `main` ブランチを保護対象にする
- 既存リポジトリを上書きしない
- 無関係な既存 remote を変更しない
