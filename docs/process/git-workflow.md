# Git Workflow

## ブランチ規則
- `main` / `master` へ直接コミットしない
- 作業ブランチは `feature` / `fix` / `refactor` / `docs` / `chore` / `test` を使う
- ブランチ名は `<type>/<short-summary-kebab-case>` とする
- 適切な既存ブランチがあれば再利用を優先する
- detached HEAD、規約外ブランチ、`main` / `master` 直作業なら、先に適切なブランチへ切り替える

## コミット粒度
- 1 コミット = 1 意図を守る
- 機能追加とリファクタを混在させない
- ロジック変更とフォーマット変更を分離する
- 自動生成物、設定変更、ドキュメント変更は別コミットを優先する
- 実装とその直接検証に必要な最小限のテストは同一コミットでよい
- 一時ファイル、環境依存ファイル、無関係な差分は含めない

## PR 規則
- 1 PR = 1 目的を守る
- レビュー可能サイズを維持し、大規模変更は段階 PR に分ける
- 本題と無関係なついで修正を混ぜない
- PR 本文は [../../.github/pull_request_template.md](../../.github/pull_request_template.md) を使う

## PR 本文の必須項目
1. 背景・目的
2. 変更内容
3. 影響範囲
4. 動作確認
5. リスク
6. レビュー観点

## XLAM 関連 PR の追加要件
- `Extract` / `Combine` の対象と前提条件を明記する
- バックアップ、確認ダイアログ、入力検証、エラーログの有無を記載する
- 破壊的変更や上書き可能性がある場合は PR 冒頭で明示する
