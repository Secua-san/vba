# docs/commit-message.md

# コミットメッセージ規約

## 目的

コミット履歴の可読性・検索性・レビュー効率を高めるため、
本リポジトリでは Conventional Commits 規約を採用する。
Codex による自動コミットも本規約に従う。

---

# 基本形式

type(scope): summary

* type : 変更の種類
* scope : 変更対象の範囲（任意だが推奨）
* summary : 変更内容の要約（日本語・簡潔・命令調）

---

# type 一覧

feat : 新機能の追加
fix : 不具合修正
refactor : リファクタリング（外部仕様変更なし）
test : テスト追加・修正
docs : ドキュメント変更のみ
chore : ビルド・設定・依存更新など
perf : パフォーマンス改善
style : フォーマット変更のみ（挙動変化なし）

---

# scope 指針

scope は変更箇所を短く示す識別子。

例：

lexer : 字句解析
parser : 構文解析
ast : AST生成
symbols : シンボル解決
inference : 型推論
diagnostics : 診断機能
lsp : Language Server機能
extension : VS Code拡張側
commands : コマンド処理
xlam : XLAM連携
vbac : vbac.wsf連携
ui : UI関連
config : 設定処理
build : ビルド関連
deps : 依存関係
ci : CI/CD
docs : ドキュメント

scope は複数可だが、可能な限り単一に保つ。

---

# summary の規則

## 必須条件

* 日本語で書く
* 命令調で書く（〜を追加、〜を修正、〜に対応 など）
* 40文字以内を目安に簡潔に
* 何をしたかが明確に伝わること

## 禁止事項

* 曖昧な表現
* 主観的表現
* 意味のない単語
* 長すぎる文章

---

# 良い例

feat(parser): プロパティプロシージャ解析を追加
fix(lsp): 不正トークン時の補完クラッシュを防止
refactor(symbols): ローカルとモジュールスコープを分離
test(parser): PtrSafe宣言のテストケースを追加
docs(agents): 自動PR安全規則の説明を追記
chore(deps): vscode-languageserverを最新版へ更新
perf(inference): 型解決結果のキャッシュ処理を追加
style(core): インデント形式を統一

# 本文（body）の書き方

詳細説明が必要な場合のみ本文を追加する。

type(scope): summary

body

---

## 本文に書く内容

* なぜ必要か（Why）
* 背景
* トレードオフ
* 制約
* 既知の制限
* 将来の改善余地

---

# 本文の良い例

feat(inference): Variant型の暗黙的昇格に対応

従来は数値演算時にLong型へ固定されていたが、
VBA特有の挙動に合わせてVariantへの昇格処理を追加。

これにより実際の実行挙動に近い型推論が可能になる。
パフォーマンス影響は軽微。

将来的には演算子単位の詳細推論へ拡張予定。

---

# フッター（footer）

必要な場合のみ使用。

用途：

* 破壊的変更の明示
* Issue番号の関連付け

例：

BREAKING CHANGE: シンボル解決順序を変更
Closes #123
Refs #456

---

# 破壊的変更の扱い

互換性を壊す変更は必ず明示する。

例1
feat(parser)!: エラー回復ロジックを変更

例2
BREAKING CHANGE: End If欠落時の許容処理を廃止

---

# コミット分割ルール

以下は分割する：

* 実装
* テスト
* リファクタ
* ドキュメント
* フォーマット
* 自動生成物
* 設定変更

例外：
実装とそれを直接検証する最小限のテストは同一コミット可。

---

# Codex 自動生成時の追加規則

Codex がコミットメッセージを生成する場合：

* summary は必ず差分の主要目的を反映する
* 「更新」「修正」だけの曖昧表現は禁止
* scope を可能な限り明示する
* 差分を見れば妥当と判断できる内容にする
* 推測ではなく実際の変更内容に基づく

---

# 推奨長さ

summary : 40文字以内
body : 必要な場合のみ
1コミット : 1意図

---

# まとめ

良いコミットメッセージは：

* 短い
* 明確
* 検索しやすい
* 差分の意図が分かる
* 将来の自分とレビューアに優しい

コミットログは「プロジェクトの歴史書」であることを意識する。
