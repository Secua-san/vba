# Skill: 安全な自動コミットおよびPR作成

## 目的
実装完了後、リポジトリ規則に従い、安全にコミットとPRを作成する。

---

## このスキルを使う場面
- 実装が完了したとき
- 修正内容をコミット単位に整理したいとき
- PR本文を一定品質で作成したいとき
- Codex に自動コミット / 自動PR を行わせたいとき

---

## 前提
- AGENTS.md の規則を最優先で遵守する
- main への直接コミットは禁止
- 変更は必ず論理単位に分割する
- 無関係な変更は含めない

---

## 実行手順

### Step 1: リポジトリ状態の確認
以下を確認する：

- 現在のブランチ名
- 未追跡ファイル
- 変更済みファイル
- main 直作業になっていないか
- リポジトリ未初期化でないか

確認結果に応じて以下を実施する：

- リポジトリ未初期化の場合  
  → Git リポジトリを初期化する
- GitHub リモートが必要で未設定の場合  
  → リポジトリ作成を提案または実施する
- main 上で作業している場合  
  → 適切な feature/fix/refactor/docs/chore/test ブランチを作成する

---

### Step 2: 差分の分類
変更内容を目的単位で分類する。

分類例：
- 新機能追加
- バグ修正
- リファクタ
- テスト追加
- ドキュメント更新
- フォーマットのみ変更
- 設定変更
- 生成物更新

以下が混在している場合は分割する：

- 機能追加 + リファクタ
- ロジック変更 + フォーマット変更
- 手動コード修正 + 自動生成物更新
- 本題と無関係なついで修正

---

### Step 3: 変更の妥当性確認
各差分について以下を確認する：

- この変更は今回の目的に必要か
- 無関係な変更が含まれていないか
- 将来のレビューで追いやすいか
- 1コミット1意図になっているか

不適切な場合：
- 差分を分割する
- 不要差分を除外する
- 自動処理を停止して要約する

---

### Step 4: 品質ゲート実行
可能な限り以下を実行する：

- lint
- build
- test

基本コマンド：
- `npm run lint`
- `npm run build`
- `npm test`

必要に応じて：
- `npm run package`

---

### Step 5: 品質ゲート結果の扱い
#### すべて成功した場合
- 次の手順へ進む

#### 失敗した場合
- 修正可能で原因が明確なら修正する
- 原因不明または高リスクなら自動コミット / 自動PR を停止する
- 停止時は以下を要約する：
  - 失敗コマンド
  - 失敗理由
  - 想定影響
  - 人間に確認してほしい点

---

### Step 6: コミットメッセージ作成
Conventional Commits を使用する。

形式：
`type(scope): summary`

例：
- `feat(parser): add procedure declaration parsing`
- `fix(lsp): avoid completion crash on invalid tokens`
- `refactor(core): split symbol resolution stages`
- `test(parser): add ptrsafe declaration cases`
- `docs(agents): clarify auto pr safety rules`

規則：
- summary は英語
- 命令形
- 簡潔
- 曖昧表現禁止
- 必要な場合のみ本文を追加する

本文に書く内容：
- なぜ必要か
- トレードオフ
- 制約
- 既知の限界

---

### Step 7: コミット作成
- 論理単位ごとにコミットする
- 不要ファイルは含めない
- 一時ファイル、環境依存ファイル、意図しない差分は除外する

原則：
- 1コミット = 1意図
- テストは直接対応する実装と同一コミット可
- フォーマット専用変更は別コミット

---

### Step 8: PR作成可否判定
以下を満たす場合のみ自動PR可：

- 仕様が明確
- 低リスク
- 差分がレビュー可能サイズ
- 禁止領域に該当しない
- 品質ゲート結果が説明可能

---

### Step 9: PR本文作成
PR本文には必ず以下を含める：

1. 背景・目的
2. 変更内容
3. 影響範囲
4. 動作確認
5. リスク
6. レビュー観点

UI 変更があればスクリーンショットを付ける。  
コマンド追加があれば使い方を簡潔に記載する。  
破壊的変更があれば明示する。

---

### Step 10: 停止条件
以下の場合は自動コミット / 自動PR を停止する：

- 仕様が曖昧
- 差分が大きすぎる
- 分割に確信が持てない
- セキュリティ影響が不明
- 認証/権限/課金/CI/CD/インフラ/DB変更を含む
- テスト失敗の原因が不明
- リポジトリ状態が不整合

停止時は勝手に進めず、状況を要約する。

---

## Step 11: CodeRabbit レビュー確認
PR 作成後は CodeRabbit のレビューを確認する。

確認対象：
- PR review
- review comment
- review thread
- 差分へのインラインコメント
- 要約コメント内の重要指摘

対応方針：
- 指摘を一覧化する
- 重複指摘をまとめる
- 明確な修正事項と、判断が必要な事項を分離する
- ノイズや誤検知の可能性があるものは即修正しない

---

## Step 12: CodeRabbit 指摘のトリアージ
各指摘について以下のいずれかに分類する：

- 採用して修正する
- 人間確認が必要
- 誤検知または非採用

### 採用して修正する条件
- 明確な不具合の可能性が高い
- 既存方針と矛盾しない
- 低リスクで修正可能
- テストや静的検証で確認できる

### 人間確認が必要な条件
- 仕様判断を含む
- 設計変更を含む
- セキュリティ影響が不明
- 既存要件と衝突する可能性がある

### 非採用とする条件
- 誤読に基づく指摘
- 既存方針に反する提案
- 過剰な最適化
- MVP 範囲外の変更提案

---

## Step 13: CodeRabbit 指摘対応の修正
- 採用した指摘のみ修正する
- 修正は論理単位でコミットする
- CodeRabbit 対応と無関係な変更は含めない
- 必要に応じてテストを追加する
- 修正後は lint / build / test を再実行する

推奨コミット例：
- fix(parser): CodeRabbit指摘の宣言解析不備を修正
- test(lsp): CodeRabbit指摘に対応する再現ケースを追加

---

## Step 14: 再レビューサイクル
- 修正コミットを push する
- CodeRabbit の再レビューを確認する
- 新規指摘があれば再度トリアージする
- 指摘が収束したら人間レビューへ引き渡す

---

## Step 15: 停止条件
以下の場合は自動対応を停止し、人間確認へ切り替える：

- CodeRabbit の指摘同士が矛盾する
- 修正により設計方針が変わる
- 2回以上修正しても同種の指摘が収束しない
- セキュリティや権限影響がある
- 指摘内容の妥当性を自動で判定できない

---

## リポジトリ作成ルール
作業に必要であれば、以下を実施してよい：

- `git init`
- 初期コミット作成
- GitHub リポジトリ作成
- `origin` 設定
- README / LICENSE / .gitignore の生成
- 初期ディレクトリ構成作成

ただし以下は必ず守る：

- 既存リポジトリを上書きしない
- 無関係な既存 remote を変更しない
- main へ直接作業しない
- リポジトリ作成理由を説明可能にする

---

## 出力時の基本姿勢
- 何をコミットしたかを要約する
- なぜその単位に分割したかを説明可能にする
- PRではレビューしやすさを優先する
- 自動化より安全性を優先する
